# LearnEVO

A ground-up study of **EvoERP** — every module, every file, every data
flow — plus a self-contained, browser-based help system built on top of
the findings. Internal i2Systems research workspace.

- **2,718** cross-linked help pages
- **88** topics, **45** recipes, **22** modules, **759** menu codes,
  **659** database tables, **1,109** UI forms, **36** glossary entries
- Full-text search (18,799 tokens) with synonyms: type `customer` to
  jump to all A/R pages, `AP-H` to jump to Print Checks, `BKARCUST` to
  jump to the customer master table
- Keyboard-first navigation (`Ctrl+K` search, `g h` home, `↑↓`, `Enter`)
- Zero external dependencies — ships as static files served by a 60-line
  local Python server

---

## Quick start

```cmd
RUN.bat
```

That's it. The script:
1. Kills any stale help-server processes from prior sessions.
2. Starts `learnevo-help\server.py` on the first free port at/after
   `8765`, bound to `127.0.0.1` only (not exposed on the LAN).
3. Opens your browser.
4. Serves every response with `Cache-Control: no-store` so CSS/JS edits
   always show up on plain reload.

Press `Ctrl+C` (or close the console window) to stop.

If `python` isn't on your `PATH`, install Python 3.10+ from python.org
and re-run.

## The stack under study

- **EvoERP** — Addsum Business Solutions' manufacturing ERP, evolved
  from DBA Manufacturing.
- **TAS Professional 7** — xBase-family 4GL runtime
  (`tp7runtime.exe`) that executes EvoERP's business logic. Compiled
  programs ship as `.RWN` (TAS 7) and `.RUN` (TAS 6).
- **Pervasive / Actian Zen (Btrieve)** — the data engine. Tables are
  `.B` files described by `.DDF` (DDF3) or encrypted `.DCY` dictionaries.
- **Nevrona ReportBuilder** — the reporting engine (`RBDsgnr.exe`,
  `.RTM` templates).
- **Delphi-style forms** — UI layouts stored as `.DFM`.
- **Java sidecar** — `EvoPVT.jar` + an `ISJAVA` task queue bridge some
  EVO operations to the JVM.

See [`docs/01-architecture/`](docs/01-architecture/) and
[`docs/02-file-formats/`](docs/02-file-formats/) for the authoritative,
citation-backed breakdown.

## How the help viewer is built

```
docs/*.md          ←  handwritten docs + autogenerated module/table pages
      │
      ▼
learnevo-help/build.py
      │   ├── walks docs/
      │   ├── flattens Markdown into learnevo-help/content/*.md
      │   └── writes learnevo-help/data/{pages,search,graph}.json
      ▼
learnevo-help/index.html  + css/style.css  + js/{app,renderer,search,marked-lite}.js
      │
      ▼
learnevo-help/server.py  →  http://localhost:8765
```

The viewer is a single-page app. On load, it fetches `data/pages.json`
(site map), `data/search.json` (token index), and lazily fetches
individual Markdown pages from `content/` as the user navigates.

To regenerate after editing `docs/`:

```cmd
cd learnevo-help
python build.py
```

## Working on this repo with Claude Code

Every long-lived rule about this workspace lives in [`CLAUDE.md`](CLAUDE.md):

- **§1 — Scope.** `C:\ISTS\` and `\\i2s109-solidcrm\` are read-only;
  work on *copies* brought into `samples/` instead.
- **§2 — Mission.** Study the actual program files as the primary source
  of truth; document findings; stay autonomous.
- **§4 — Conventions.** Naming, folder layout, status tags (`draft` /
  `partial` / `verified` / `open-questions`), citation rules.
- **§5 — Autonomy protocol.** After finishing a documentation pass,
  immediately pick the next unknown from `research/OPEN_QUESTIONS.md`.
- **§6 — Bug-fix tracking.** Every bug attempt (success or failure)
  must be logged in `BUGS.md` before moving on. No repeating failed
  fixes.
- **§7 — Launcher upkeep.** Any startup-related change must update the
  `.bat` / `.ps1` launchers in the same commit.
